# ERROR_PATTERNS.md — подтверждённые паттерны ошибок n8n

Источники: собственные сессии отладки (агробот март 2026, MAX-бот июнь 2026,
Router TEST июль 2026) и константы из SKILL.md. Не добавлять паттерн без
воспроизведённого кейса.

## 1. Redis GET затирает $json целиком

**Симптом:** после Redis GET пропадают поля контекста (chat_id и др.), ноды ниже
получают undefined.
**Причина:** Redis-нода перезаписывает item полностью, а не мержит.
**Фикс:** восстанавливать контекст ссылкой на исходную ноду:
`$('NodeName').first().json` (или `.item.json` на путях с несколькими item).

## 2. Анти-паттерн "Restore After X"

**Симптом:** workflow разрастается парами нод "запрос к БД + Code-нода восстановления
контекста" (наблюдалось 8 штук в одном роутере).
**Причина:** следствие паттерна 1 — каждое обращение к Postgres/Redis затирает item.
**Фикс:** не протаскивать контекст через цепочку, а ссылаться на исходные ноды
напрямую в выражениях. Осторожность — только на путях с несколькими item.

## 3. Выражение без "=" в начале поля

**Симптом:** в поле видно `{{ $json.field }}`, а в рантайме — literal-текст или
undefined.
**Причина:** без префикса `=` n8n трактует содержимое поля как статический текст.
**Фикс:** `={{ $json.body.text }}` — знак `=` переключает поле в режим выражения.

## 4. Поле не принимает выражения вовсе

**Симптом:** ошибка вида `Invalid User ID: "=". Must be a number` (MAX-нода, поле
User ID).
**Причина:** отдельные поля нод валидируют тип до резолва выражения.
**Фикс:** для теста — захардкодить значение; для прода — проверить, есть ли у ноды
альтернативный параметр/режим, принимающий выражение.

## 5. Memory sub-node требует физического соединения

**Симптом:** `connect "Redis Chat Memory" to "<Node>"`, в стеке
`getInputConnectionData`. Ошибка воспроизводится и в рантайме
(NodeOperationError), не только в тестовом режиме.
**Причина:** sub-node (Memory у AI Agent) резолвит выражения в контексте своего
физического подключения; ссылка `$node["X"]` на неподключённую ноду ненадёжна.
**Фикс:** ключ сессии должен приходить из данных, доступных в цепочке физического
соединения sub-node. Ссылки на произвольные ноды из sub-node не использовать.

## 6. Выражения в sub-node резолвятся только к первому item

**Симптом:** в sub-node `{{ $json.name }}` всегда возвращает значение первого item.
**Причина:** особенность контекста sub-node.
**Фикс:** учитывать при проектировании; при необходимости агрегировать данные до
входа в родительскую ноду.

## 7. Postgres: сравнение chat_id падает или не находит строки

**Симптом:** запрос с chat_id из Telegram/MAX не матчится или даёт ошибку типа.
**Причина:** chat_id приходит числом за пределами int4 / строкой; тип колонки bigint.
**Фикс:** явный каст в запросе с queryReplacement:
`SELECT ... WHERE chat_id = $1::bigint` + `options.queryReplacement: ={{ [$json.chat_id] }}`.

## 8. HTTP Request нода не резолвит выражения в JSON body

**Симптом:** в теле запроса уходят нерезолвленные `{{ }}` или пустые значения.
**Фикс:** Code-нода с `this.helpers.httpRequest()` вместо HTTP Request ноды.

## 9. Retry on Fail + Continue on Fail несовместимы

Одновременное включение не работает (bug #9236). Выбирать одно.

## 10. .first() в Code-ноде

`$('Node').first()` требует режим `runOnceForAllItems`; в `runOnceForEachItem`
падает с `Can't use .first() here`.

## 11. Gemini 2.5 Flash с reasoning: два parts

`parts[0]` = thoughtSignature, `parts[1]` = целевой JSON. Парсить `parts[1]`,
не `parts[0]`.
